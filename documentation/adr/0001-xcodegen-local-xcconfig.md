# ADR-0001: Use XcodeGen with a gitignored `Local.xcconfig`

- **Status**: Accepted
- **Date**: 2026-04-23

## Context

A macOS app needs an Xcode project. `.xcodeproj` bundles are notoriously merge-unfriendly and commonly churn on unrelated developer-local settings (signing team, derived data paths). The project also requires a real Apple Developer Team ID at signing time for the Data Protection Keychain entitlement (see [ADR-0004](0004-data-protection-keychain.md)), and contributors shouldn't commit their Team IDs to a shared repo.

## Decision

`project.yml` at the repo root is the **source of truth** for the Xcode project. `StockAlerts.xcodeproj/` is generated by [XcodeGen](https://github.com/yonaskolb/XcodeGen) (`brew install xcodegen`; `xcodegen generate`) and is gitignored.

A second file, `Local.xcconfig`, supplies `DEVELOPMENT_TEAM` per contributor and is also gitignored. A committed `Local.xcconfig.example` documents the required format. `project.yml` references `Local.xcconfig` via `configFiles.Debug` / `configFiles.Release`.

## Consequences

- Every file add/remove/rename requires a follow-up `xcodegen generate` before the build system sees it.
- First-time contributors must `cp Local.xcconfig.example Local.xcconfig` and fill in their Team ID before the project will generate / sign cleanly (documented in `README.md`).
- Signing team is not leaked into the repo; each developer (or CI) provides their own.
- Entitlements, bundle ID, deployment target, and scheme configuration all live in one reviewable YAML file.

## Alternatives considered

- **Commit `StockAlerts.xcodeproj`**: rejected because of merge-conflict churn, Team ID leakage, and the cost of manually editing generated XML.
- **Tuist**: comparable capability but heavier dependency and Swift-based manifest; overkill for a single-target macOS app.
- **Swift Package Manager only**: insufficient. SPM alone cannot produce a macOS `.app` bundle with `MenuBarExtra`, entitlements, and a `Settings` scene.
